# Skill Review Gates

规则本体在 principles.md，本文件只提检查问句。

执行口径：所有适用项都是必须通过的 gate；不适用标 N/A 并说明理由；不得用“基本符合/后续再改”放行；无法满足的适用项标 blocked 并说明缺什么。

## 范围

- 是否只改了用户点名的范围及其强依赖？
- 是否未顺手重构相邻 skill？

## 最小改动

- 有无可以不改的重排或重写？
- 是否优先编辑已有文件而非新建平行文件？

## 语言

- 正文是否中文默认？是否表达精准简洁？
- 每条规则是否可执行、可检查？
- 是否无 no-op（模型默认就会做的）行为描述？

## 冗余

- 同一条规则是否只在一处出现？
- controller 是否未复制子 skill 的阶段细则？

## 冲突

- description 是否不会误触发相邻 skill？
- 读写与用户确认规则是否与既有 workflow 不冲突？
- 现存规则之间是否自相矛盾

## 命名

- name 是否 kebab-case 且唯一？
- 是否为活动名而非业务对象名？
- 标题、目录、name、description 是否指向同一职责？

## invocation 与判据

- invocation 选择是否有依据？
- 双 harness 是否成对同步？
- 每个工作单元的判据是否可证伪：敷衍与诚实的产出是否不同形？确认式判据（“确认无误”）视为不合格。
- 判断类规格是否细到可裁决争议、粗到不可照抄生成？
- 禁止式规则能否正向化？

## 脚本边界

- 脚本是否只做结构、格式、路径、字段等机械检查？
- 语义 verdict 是否归模型/人？
- 机械可判的细则是否已下沉脚本/schema，而非留在判断文本里？

## 输入与产物必要性

- skill 要求的每项读取是否任务所需、只取所需片段？
- 每个产物（含中间产物）是否说得出用途与服务对象？
- 是否无用户未要求的 output 文件？
- 质量所需的读取、查证、验证是否未因省 token 被砍？
